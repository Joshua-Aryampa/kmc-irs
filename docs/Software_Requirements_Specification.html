1. Introduction

1.1 Purpose

This Software Requirements Specification (SRS) defines the complete functional and non-functional requirements for the Incident Reporting System (IRS). The document describes the system as implemented and serves as the authoritative reference for acceptance testing, maintenance, training, and future enhancements. Its intended audience includes project stakeholders, quality assurance personnel, developers, system administrators, and end users in supervisory roles.

1.2 Scope

The IRS is a web-based application that digitizes the production-line incident reporting form and its multi-stage approval workflow. The system enables authorized personnel to create, submit, verify, approve, reject, and resubmit incident reports; attach photographic evidence; maintain an immutable audit timeline; receive email notifications at workflow transitions; search and filter incidents on a dashboard; export filtered data to CSV; and generate PDF reports for closed incidents. Administrative functions include user provisioning and manual reassignment of verifiers and approvers on open incidents.

The scope of this document covers the application layer delivered as a Django web project. Deployment infrastructure (reverse proxy, TLS certificates, operating system hardening, enterprise identity provider integration) is referenced only where it constrains or interfaces with the application. The IRS does not include shop-floor equipment integration, automated severity alerting by classification, or incident reopening after closure.

2. Overall Description

2.1 Product Perspective

The IRS is a self-contained web application that replaces paper-based or ad hoc electronic incident reporting for a vehicle manufacturing plant production department. It operates as a centralized system of record for incident data and workflow state. Users interact through standard web browsers on desktop computers, tablets, and mobile phones. The application persists data in a relational database, stores uploaded images on the file system, and sends notifications through a configurable SMTP-compatible email backend.

The system integrates externally only with: (a) an SMTP mail server or console email backend for notifications; (b) a relational database (SQLite for development, PostgreSQL recommended for production); and (c) optional future enterprise identity services (not implemented in the current release; authentication is username/password via Django session management).

2.5 Design and Implementation Constraints

  • The application shall be implemented using the Django web framework and server-rendered HTML templates.
  • User interface styling shall use CSS with responsive layout support for mobile and desktop viewports.
  • Authentication shall use Django’s built-in user model extended with role and designation attributes.
  • Workflow state transitions shall be enforced in server-side service logic, not solely in client scripts.
  • Incident identifiers shall be assigned atomically using a year-scoped database sequence to prevent duplication.
  • PDF generation shall use the xhtml2pdf library from HTML templates.
  • Maximum photograph upload size and count shall be enforced server-side via configurable settings.

2.6 Assumptions and Dependencies

  • Each production scene location has at least one active Supervisor and one active Shop Floor Manager assigned, or worker-submitted reports cannot be routed.
  • Reporter, verifier, and approver for any given incident are three distinct user accounts.
  • Users possess valid email addresses stored in their profiles for notification delivery.
  • Plant management accepts that closed incidents cannot be reopened through the system.
  • Severity is assigned by the verifier at verification time and is informational; it does not trigger automated alerts.
  • Network connectivity is available at the time of report submission; offline capture is out of scope.
  • IT Admin maintains user accounts and location assignments through the application’s admin interface.

3.4 Workflow and Routing

3.4.1 Incident Status Model

StatusDescription
DraftReport created but not submitted.
Pending verificationSubmitted and awaiting assigned verifier.
Returned to reporterSent back for correction with comment.
Pending approvalVerified and awaiting assigned approver.
Returned to verifierRejected by approver; awaiting verifier action.
ClosedApproved and finalized; no further edits permitted.

3.4.2 Routing Matrix

On submit or resubmit, the system shall assign verifier and approver automatically according to reporter role and scene location:

Reporter RoleVerifier AssignedApprover Assigned
Worker at location LSupervisor at location LShop Floor Manager at location L
Supervisor at location LShop Floor Manager at location LDirector of Production
Shop Floor Manager at location LDirector of ProductionCEO

If no active user exists for a required role and location, submission shall fail with an instruction to contact IT Admin. If reporter, verifier, and approver would resolve to fewer than three distinct users, submission shall fail with a role-separation error.

3.4.3 Workflow Actions

IDPriorityRequirement
FR-WF-001MustOn first successful submit, the system shall assign a unique incident identifier and set status to Pending verification.
FR-WF-002MustOn resubmit after return, the system shall retain the same incident identifier and reassign the same verifier and approver pair resolved at submission.
FR-WF-003MustThe system shall flag a submission as late when submit time is strictly greater than incident date/time plus 30 minutes (configurable via INCIDENT_LATE_MINUTES). Submissions at exactly 30 minutes shall not be late.
FR-WF-004MustLate submissions shall require a non-empty late reason on both client and server validation.
FR-WF-005MustOn resubmit, previously assigned severity shall be cleared until the verifier assigns it again.
FR-WF-006MustThe assigned verifier shall verify an incident only when status is Pending verification.
FR-WF-007MustVerification shall require selection of severity from: Insignificant, Minor, Moderate, Major, Catastrophic.
FR-WF-008MustSuccessful verification shall set status to Pending approval and record verifier confirmation timestamp.
FR-WF-009MustThe verifier shall reject to reporter with a mandatory comment, setting status to Returned to reporter.
FR-WF-010MustThe assigned approver shall approve only when status is Pending approval.
FR-WF-011MustApproval shall set status to Closed, record approver confirmation and closed timestamps, and prohibit further modification.
FR-WF-012MustThe approver shall reject to verifier with a mandatory comment, setting status to Returned to verifier.
FR-WF-013MustWhen status is Returned to verifier, the assigned verifier may forward to reporter with a mandatory comment, setting status to Returned to reporter.
FR-WF-014MustVerifier and approver shall not edit reporter-entered report fields; they may only perform workflow actions defined above.
FR-WF-015MustClosed incidents shall not be reopened through the application.

3.4.4 Incident Identifier Format

On first submit, the system shall generate an incident identifier using the pattern: {PREFIX}{MM}/{DD}-{YYYY}-{SEQ} where PREFIX is an organization-configured constant (default: organizational department prefix ending with a dot); MM/DD-YYYY derive from submission timestamp; SEQ is a five-digit zero-padded sequence number resetting each calendar year.

Example pattern: ORG.DPN.06/24-2026-00001

4. External Interface Requirements

4.1 User Interfaces

The application presents server-rendered HTML pages with responsive CSS layout. Primary screens are: Login, Dashboard, My Queue, Incident Create/Edit form, Incident Detail, User List, User Create/Edit, and Reassign. Forms use accessible labels, inline validation messages, and role-appropriate action buttons. The interface shall be usable on viewport widths from 320px (mobile) to large desktop displays.

4.2 Hardware Interfaces

None specific. Client devices require a web browser and optional camera or file picker for photograph upload. No direct interface to shop-floor sensors, PLCs, or barcode scanners is required.

4.3 Software Interfaces

InterfaceDescription
Relational databaseSQLite (development) or PostgreSQL (production) via Django ORM.
File systemMedia root directory for incident photo storage organized by year/month.
SMTP emailDjango email backend; configurable to console, SMTP, or other Django-supported backends.
PDF enginexhtml2pdf rendering HTML templates to PDF byte stream.
Static assetsCSS and JavaScript served from application static directory; web fonts loaded from Google Fonts CDN.

4.4 Communications Interfaces

Clients communicate with the application over HTTP or HTTPS. Email notifications use SMTP (typically port 587 with TLS). No WebSocket, message queue, or REST API for external consumers is provided in the current release.

6. Data Requirements

6.1 Plant Locations

The system shall maintain a fixed set of active production scene locations. Initial seed data includes six areas: Trim and Final Assembly, Body Shop, Chassis Line, Paint Shop, Machine Shop, and Quality Inspection/Test (QIT). Locations are referenced by incident reports and by Supervisor/Shop Floor Manager assignments.

6.3 Data Retention

The system shall retain closed incident records, timeline entries, photographs, and notification logs indefinitely unless superseded by an organizational records-management policy implemented outside this application. No automatic purge is performed by the current version.

7. Other Requirements

7.1 Regulatory and Compliance

The IRS supports production department operational reporting requirements. Organizations may map retained incident records to internal ISO 9001 or occupational health and safety documentation practices. The system does not automatically generate regulatory filings to external authorities.

7.2 Localization

The current release is English-only. All dates and times display in the configured application time zone. Future localization is out of scope for version 1.0.

7.3 Operational Requirements

  • IT Admin shall verify at least one active Supervisor and Shop Floor Manager per location before production rollout.
  • SMTP credentials and IRS_BASE_URL shall be configured before email notifications are relied upon in production.
  • A management command (seed_data) is provided for development/demo population of locations, users, and ID sequence; it shall not be run against production databases without controlled procedures.
