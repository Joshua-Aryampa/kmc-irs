1. Introduction

1.1 Purpose

This Software Design Document (SDD) describes the architecture, component structure, data model, interfaces, and detailed design of the Incident Reporting System (IRS) as implemented. It translates the requirements defined in IRS-SRS-001 into a technical blueprint for developers, maintainers, quality assurance staff, and IT operations personnel. The document explains how the system satisfies functional and non-functional requirements through concrete design choices, module boundaries, and processing flows.

1.2 Scope

The SDD covers the complete Django web application delivered in the kmc-irs/ project directory, including the accounts and incidents applications, configuration layer, HTML templates, static assets, service modules, and management commands. Deployment infrastructure (load balancers, TLS termination, operating system configuration) is described at a high level only where it affects application design. Out of scope: enterprise identity provider integration (future), shop-floor device integration, and automated alerting by severity or classification.

2. System Architecture Overview

2.1 Architectural Style

The IRS follows a monolithic, server-rendered web application architecture built on Django 5.x. The design separates concerns into four logical layers:

  1. Presentation layer — HTML templates, CSS, and minimal client-side JavaScript for form UX (witness rows, photo previews, late-reason visibility, dashboard auto-filter, verify-button gating).
  2. Application layer — Django views handling HTTP requests, form binding, authorization checks, and response rendering.
  3. Domain/service layer — Stateless Python modules (workflow, routing, notifications, export, pdf) implementing business rules in transactional units.
  4. Persistence layer — Django ORM models mapped to SQLite (development) or PostgreSQL (production), plus file-system storage for uploaded images.

4.2 Accounts Application

4.2.1 User Model (accounts/models.py)

Extends AbstractUser with:

  • designation — job title displayed on declaration and reports.
  • role — enum: Worker, Supervisor, Shop Floor Manager, Director of Production, CEO, Admin.
  • assigned_location — foreign key to Location; required for Supervisor and Shop Floor Manager only.

Method can_report() returns true for Worker, Supervisor, and Shop Floor Manager. Model clean() enforces location assignment rules consistent with admin form validation.

4.3.1 Draft vs Submit Form Processing Pattern

Create and edit views instantiate two logical form instances from the same POST data:

  1. draft_form — IncidentForm(..., submitting=False) for persisting partial data.
  2. submit_form — IncidentForm(..., submitting=True) when action is submit, enforcing full validation including declaration, classifications, and late reason.

On validation failure during submit, the draft is saved first (including witnesses and new photos) so the user does not lose uploaded files or entered data. This design satisfies FR-RPT-004 from the SRS.

4.6 Classifications Configuration (incidents/classifications.py)

Declarative dictionary INVOLVE_GROUPS maps each Involves category to: display label, model boolean field name, list of (field_name, label) classification pairs, and Others boolean/text field names. Templates and form validation iterate this structure, avoiding hard-coded duplication across HTML and Python.

4.7 Service Layer

4.7.1 Workflow Service (incidents/services/workflow.py)

Each public function is decorated with @transaction.atomic. Internal helper _log creates TimelineEntry records. Late submission helpers _incident_datetime, is_late_at_submission, and compute_late use configured INCIDENT_LATE_MINUTES and plant time zone.

FunctionPreconditionsPostconditions
submit_incidentReporter actor; Draft or Returned status; late reason if lateID assigned; routing resolved; status → Pending verification; notifications sent
verify_incidentVerifier actor; Pending verification; severity non-emptySeverity stored; status → Pending approval
reject_verificationVerifier actor; non-empty commentStatus → Returned to reporter; return_comment set
approve_incidentApprover actor; Pending approvalStatus → Closed; closed_at set
reject_approvalApprover actor; non-empty commentStatus → Returned to verifier; pending_approver_comment set
forward_to_reporterVerifier actor; Returned to verifier statusStatus → Returned to reporter
reassignAdmin actor; open incident; valid replacement userVerifier or approver updated; timeline logged

4.7.2 Routing Service (incidents/services/routing.py)

resolve_verifier_approver(reporter, scene_location_id) implements the routing matrix from SRS Section 3.4.2 using _one_user to select the first active user matching role and location. _validate_role_separation ensures three distinct user IDs.

generate_incident_id locks the IncidentSequence row for the submission year, increments last_sequence, and formats: {PREFIX}{MM}/{DD}-{YYYY}-{SEQ:05d}.

eligible_reassignments(incident, role_kind) returns queryset of valid replacement users excluding reporter, verifier, and approver IDs.

4.7.3 Notifications Service (incidents/services/notifications.py)

Each notify function builds subject/body strings and calls _send, which creates a NotificationLog row, invokes Django send_mail, and updates log status to SENT or FAILED with error message. Deep links use IRS_BASE_URL/incidents/{pk}/.

4.7.4 Export Service (incidents/services/export.py)

incidents_csv(queryset) returns HttpResponse with Content-Disposition attachment and ten-column CSV header matching dashboard-visible attributes.

4.7.5 PDF Service (incidents/services/pdf.py)

render_incident_pdf renders HTML template with base64-encoded photo data URIs, passes output through pisa.CreatePDF, returns raw PDF bytes for HTTP response.

4.8 Context Processors

incidents/context_processors.nav_counts injects queue counts into every template for authenticated users. On login, session key show_queue_alert triggers one-time banner display then is popped from session.

5.3 Indexing and Integrity

  • incident_id unique constraint prevents duplicate public identifiers.
  • Foreign keys use PROTECT on Location and User references where deletion would orphan active workflow assignments.
  • CASCADE on Witness, Photo, Timeline, NotificationLog ensures incident deletion removes dependent records (admin operation only).
  • Default queryset ordering: incidents by -created_at; timeline by created_at ascending.

6. Workflow and Business Logic Design

6.1 State Machine

Incident status is a finite state machine with six states. Transitions are permitted only through workflow service functions invoked by authorized views. Invalid transitions raise WorkflowError, surfaced to users as flash messages.

Current StateEvent / ActorNext State
DraftSubmit / ReporterPending verification
Returned to reporterResubmit / ReporterPending verification
Pending verificationVerify / VerifierPending approval
Pending verificationReject / VerifierReturned to reporter
Pending approvalApprove / ApproverClosed
Pending approvalReject / ApproverReturned to verifier
Returned to verifierForward / VerifierReturned to reporter
Closed—Terminal (no transitions)

6.2 Submit Sequence

Reporter → [Submit POST] → View validates forms/photos
    → submit_incident()
        → compute_late()
        → generate_incident_id() [if first submit]
        → resolve_verifier_approver()
        → UPDATE incident (status, assignments, clear severity)
        → INSERT timeline (SUBMITTED, optional LATE_FLAGGED)
        → notify_submitted() → Email to verifier
    → Redirect to incident_detail

Figure 6-1. Submit incident processing sequence

6.3 Late Submission Algorithm

incident_dt = timezone.make_aware(combine(incident_date, incident_time))
submitted_at = timezone.localtime()
is_late = submitted_at > incident_dt + timedelta(minutes=INCIDENT_LATE_MINUTES)

Equality at exactly the threshold minute boundary is not late. Validation occurs in IncidentForm.clean(), client-side JavaScript before submit, and submit_incident as final authority.

6.4 Routing Matrix Implementation

Implemented as conditional branches in resolve_verifier_approver keyed by reporter.role and scene_location_id. Director and CEO approvers/verifiers are plant-wide (no location filter). Worker-path roles filter by incident scene location.

8.2 Email Interface

Django email backend abstraction supports console (development) and SMTP (production). Message format is plain text with incident metadata and HTTPS/HTTP link. Notification types: SUBMITTED, VERIFICATION_REJECTED, VERIFIED, APPROVAL_REJECTED, CLOSED.

8.3 File Storage Interface

Uploaded images stored under MEDIA_ROOT/incidents/{YYYY}/{MM}/ with original filename and MIME recorded in database. DEBUG mode serves media via Django static helper; production requires web server or object storage configuration external to this document.

8.4 Database Interface

All persistence through Django ORM. No raw SQL in application code. Migrations in accounts/migrations/ and incidents/migrations/ version the schema.

9. Security Design

9.1 Authentication

Django session middleware maintains authenticated state after login. Session cookie is HTTP-only by default in Django. Passwords hashed via PBKDF2 (Django default). Login redirect: dashboard; logout redirect: login.

9.2 Authorization Model

Role-based access control (RBAC) combined with assignment-based checks (verifier/approver/reporter foreign keys). Views enforce:

  • Dashboard queryset scoped by incidents_for_user.
  • Detail view blocked via user_can_view_incident (HTTP 403).
  • Workflow POST handlers verify acting user matches assigned verifier or approver inside service layer.
  • Admin-only routes check user.role == ADMIN.

9.3 Input Validation and CSRF

  • All POST forms include {% csrf_token %}.
  • File uploads validated for MIME type, size, and count server-side.
  • Comment fields enforce minimum length to reduce empty rejections.
  • User-generated HTML is escaped by Django template engine (no raw HTML injection in standard views).
